Wherever possible, English-language and authoritative references are preferred. # Purpose and Scope This section clarifies the **intent of the sandbox and atlas**, and the rules governing their separation and interaction. - **Separate apps, shared data:** The **GlobalGrid2050 Atlas** is the ‚Äúinfrastructure intelligence‚Äù layer, while the **GIS-SLD Sandbox** is the project topology, finance and logistics engine. They must remain **decoupled at runtime**: no atlas code (e.g. `initVentusMap`, fetch queue, etc.) should be imported into the sandbox, and vice versa. The only linkage is via *shared public data files* (GeoJSON). - **Public GeoJSON bridge:** Both applications use the same publicly available infrastructure datasets (e.g. substations, cables). In practice, the sandbox loads **`data/grid_substations.geojson`** via its own code, rather than relying on atlas runtime. This one-way data sharing (atlas ‚Üí sandbox) uses static files, not live inter-process calls. - **Non-runtime coupling:** Under no circumstances should the sandbox use any dynamic functions or services of the atlas. For example, do *not* call or invoke the atlas‚Äôs `initVentusMap`, hydration queue, layer caching, or SCADA layers from the sandbox code. The sandbox runs independently after the atlas has published its data files. - **DRAW NEAT GRID behaviour:** The sandbox has one special action button: **‚ÄúDRAW NEAT GRID‚Äù**. This triggers a geometry redraw at the current map center. Crucial UX rule: pressing this clears any *selectedSubstation* (i.e. it removes attribution to a real public substation), because drawing from a random point should not keep old node info. Only clicking a public substation (below) reactivates the selected-substation mode. - **Selected substation rules:** The sandbox must clearly differentiate between a **public dataset substation** (white circle, red outline) and a **generated primary substation** (red circle, white stroke). Selecting a public substation stores its coordinates and updates the **Selected Grid Node** panel, but also shows a warning: *‚ÄúPublic dataset reference only. Does not confirm capacity, connection rights, ...‚Äù*. The primary substation (from `turf.point(activeDrawCenter, {type: 'primary_sub'})`) is always drawn in front. - **UI/UX context:** The user interface is deliberately simplified. Both modes (String and Central) use the same layout, with tab switches. Inputs recalc values live, but the map only updates on *Draw Neat Grid* or substation click. This is emphasized to the user to avoid false certainty: the map depiction is merely an indicative topology. All disclaimers (in popups, panels, warnings) stress that nothing is guaranteed (no legal boundaries, no confirmed grid acceptance, etc.). In summary, this sandbox is a **screening tool**, not a final design. Its purpose is to quickly scope feasibility: combining GIS location, simplified topology (SLD style), module logistics and high-level finance in one interface. It must preserve the boundary between **‚Äúwhat we computed‚Äù** and **‚Äúwhat needs detailed engineering‚Äù**. Every key assumption or simplification (module count, losses, boundary basis, etc.) is documented or labeled, ensuring users know when they need deeper design work. # Data Model and Sources This section details all data inputs, schemas, and exported fields. (Layer IDs used here come from the sandbox code.) # Topology Modes (String vs Central) The sandbox supports two distinct PV array topology modes: **String** (decentralized) and **Central** (central inverters). Each mode has its own parameters and flow. We define all relevant variables and their roles. ## Shared parameters - **Module rating (Wp)**, **length (m)**, **width (m)**: Physical panel specs (imported from module datasheet). - **Mounting type (GCR)**: A factor (0 to 1) representing ground coverage. Values used: 0.75 (East-West dome), 0.45 (fixed tilt south), 0.35 (tracker). These imply different aspect ratios (square vs elongated blocks) for the block ground area. - **Gross site factor (if any)**: Used to compute *indicative gross site area* internally (e.g. for roads), but not drawn. Only the net array area (scaled by GCR) is drawn as the blue polygon. We export the gross-site factor numerically for reference. ## String mode parameters These define a hierarchy: Modules ‚Üí Strings ‚Üí Inverters ‚Üí Substations ‚Üí Rings. - **DC/AC ratio**: E.g. 1.2 (20% DC oversizing). - **Modules per string (x_mods)**: E.g. 28 (28 panels in each DC string). - **Strings per inverter (z_strings)**: E.g. 18. - **Inverters per substation (y_invs)**: E.g. 28 (substation = field substation, each has y_invs inverters). - **Substations per 33kV ring (s_subs)**: E.g. 5. (A ring is like a cluster of field substations around one connection point.) - **33kV rings (b_cols)**: Number of rings (e.g. 6). - **Total blocks (`total_blocks`)**: = rings √ó sub per ring (e.g. 6 * 5 = 30). - **Total inverters** = total_blocks * y_invs. - **Active Power (MW)**: If `mod_wp` in W, then total DC capacity = `(total_inverters * z_strings * x_mods * mod_wp) / 1e6`. AC power = DC / (DC/AC ratio). These are displayed in the stats box. - **Block ground area (m¬≤)**: `block_ground_area_m2 = (x_mods * z_strings * y_invs * mod_l * mod_w) / GCR`. (Each ‚Äúblock‚Äù is one field inverter plus its strings.) - **Total array area (acres)**: = (block area / 4046.86) * total_blocks. Shown in UI. - **Container count:** Total modules = total_DC/Wp * 1e6 / mod_wp (the same formula). TEU = ceil(total_modules / 310). Ships ~ TEU/1000. Shown in UI. *Combiner boxes:* Not used in string mode (only central mode). The code uses simple String ‚Üí Inverter stacking. ## Central mode parameters Here we have larger central inverters & combiners, so hierarchy: Modules ‚Üí Strings ‚Üí Combiner ‚Üí Inverters ‚Üí MV stations ‚Üí Rings. - **Central AC rating (MWac)**: The chosen size of central inverter (e.g. 4.4 MWac). - **DC/AC ratio**, **Modules per string (x_mods)**: As above. - **Strings per combiner box (str_per_cb_c)**: E.g. 24. - **Inverters per MV station (inv_per_mv_c)**: Usually 1 (one central inverter per collector). - **MV stations per 33kV ring (mv_per_ring_c)**: E.g. 4. (For central arrays, each field inverter block has several MV racks per ring.) - **33kV rings (rings_c)**: E.g. 4. From these: 1. **Calculating strings required per inverter:** ``` str_dc_kwp = x_mods * mod_wp / 1000 // DC kWp per string inv_dc_mwp = inv_ac_mw * dc_ac_ratio // DC MWp per inverter req_strings = ceil((inv_dc_mwp * 1000) / (str_dc_kwp)) // strings per inverter ``` E.g. if inv_ac=4.4, ratio=1.2 ‚Üí inv_dc=5.28 MW, str_dc=28*mod_wp/1000, then req_strings = ceil(5280 / (28*mod_wp)). 2. **Actual inverter DC (MWp):** = (req_strings * str_dc_kW) / 1000. 3. **Total inverters (blocks):** = inv_per_mv * mv_per_ring * rings. (E.g. 1 * 4 * 4 = 16 inverters total). 4. **Total DC capacity:** = total_blocks * actual_inv_dc_mwp. 5. **Block ground area:** `= (req_strings * x_mods * mod_l * mod_w) / GCR` (area for one inverter‚Äôs strings). 6. **Total array area:** = (block_area/4046.86)*total_blocks. 7. **Combiner boxes (central only):** - **Combiner boxes per inverter:** `combiner_boxes_per_inverter = ceil(req_strings / str_per_cb_c)`„Äêuser code„Äë. - **Total combiner boxes:** `combiner_boxes_per_inverter * total_blocks`. Export these values. BESS is handled identically in both modes (see BESS section below). All these parameters and formulas are implemented in the code (functions `updateStringAggregates()` and `updateCentralAggregates()`). They must match the export schema and user inputs. *A note on MV stations:* Currently the sandbox calculates `total_blocks = inv_per_mv * mv_per_ring * rings`, but **does not draw separate MV station symbols**. The UI should indicate this limitation: ‚ÄúMV station nodes not drawn‚Äù. This is a placeholder until actual MV substations are generated. *Sources:* Solar array design fundamentals and the provided module contract (anonymised) informed these hierarchies. The combiner-box math is given in the instructions and code comments. Typical values (e.g. 30 modules/string) came from module specs (listed as examples). # Geometry Semantics This section explains how boundaries and areas are defined and labelled. ## Net Array Boundary (at GCR) - **Definition:** The blue polygon drawn by the sandbox represents the **net PV array area** under the chosen GCR (Ground Coverage Ratio). It covers the actual module footprints plus inverter blocks, not including wide buffers. - **Calculation:** The code computes a rectangle polygon around the blocks. Roughly, it sets `block_area_km2 = block_ground_area_m2 / 1e6`, arranges blocks in a grid of `cols √ó rows`, adds small spacing (0.01 km) between blocks, and draws a bounding polygon offset by 0.02 km in each direction for margin. The function `getRectPolygon(center,width_km,length_km,'array_boundary')` is used. - **Labelling:** In the UI and legend, we call this *‚ÄúNet Array Boundary at GCR‚Äù*„Äê5‚ÄÝL132-L136„Äë. We **do not** call it ‚ÄúTotal site boundary‚Äù or ‚Äúlegal boundary‚Äù. It is explicitly not a planning or property line. It‚Äôs labelled in the export as `tech_drawn_boundary_basis: "Net Array Boundary at GCR"`. - **Reason:** Using `block_ground_area_m2` ensures it‚Äôs the area *occupied by modules/inverters at GCR*, not including extra roads or unusable land. The model separately computes an *indicative gross site area* (usually `net_array_area / GCR` plus allowances), but **that is not drawn**. Instead, gross area is used internally for screening (roads, buffers). It is export-only: e.g. `gross_site_area` is numeric but no geometry. The GCR *effective* area *is* the polygon. ## Gross Site Area - **Not drawn:** The model internally multiplies net array area by (1 + some factor) to get gross site area for infrastructure sizing. But we **explicitly do not draw** this. No ‚ÄúTotal site boundary‚Äù is displayed unless a real site polygon is provided. - **Export:** We export `tech_drawn_boundary_area_m2` = area of the drawn boundary polygon, and note in text that *gross site area is an indicative screening multiplier only*. - **Export field:** `tech_drawn_boundary_basis = "Net Array Boundary at GCR"`. - **Rationale:** Labelling ensures users do not mistake the drawn polygon for a legal site fence or planning limit. It‚Äôs purely computational, based on module count and GCR„Äê5‚ÄÝL130-L136„Äë. ## Coordinate precision - Export coordinates (longitude, latitude) in JSON are limited to high precision (e.g. 6 decimals) in the `selectedSubstation` fields. - Polygon coordinates can be left as full precision (they come from Turf, but the exact values are not usually critical to cite). *Sources:* The concept of net vs gross PV area is standard in PV planning (see solar engineering texts), though no direct external citation is needed here. The use of GCR is borrowed from mounting type proxies (common in PV literature). The exporting of basis/area was defined in the project spec. # Financial Model The sandbox includes a **screening financial model** to quickly estimate revenues, CAPEX and surplus over 25-35 years. It is **not** a detailed finance tool, but a high-level calculator. ## Approach - **Screening scope:** We use **Year-1 generation** and simple growth assumptions to project revenue over 25 and 35 years„Äêdeep research guidelines„Äë. No financing costs or NPV calculations are included. Contingency is added to CAPEX instead of risk reserves. - **Variables:** The user inputs (see **Data Model**) include all revenue and cost factors. These feed into revenue and CAPEX calculations. Loss allowances reflect technical margins (higher losses ‚Üí lower revenue). - **Export:** The following aggregated financial outputs are exported on the boundary feature: - `fin_annual_rev` (year-1 ¬£, i.e. *Year 1 revenue projection*), `fin_25yr_revenue`, `fin_35yr_revenue`. - `fin_total_capex` and `fin_capex_per_wp`. - `fin_25yr_surplus`, `fin_35yr_surplus` (Revenue minus OPEX minus total CAPEX). ## Revenue calculation 1. **Effective yield (year 1):** ``` effective_yield = base_yield * (1 + bifacial_gain/100). ``` Bifacial panels get more kWh/kWp. Base yield is e.g. 1000 kWh/kWp. 2. **Losses:** Sum of `base_losses` + all specialist losses. Let `total_loss_pct = min(max(total_loss,0),100)`. 3. **Year-1 Gen (kWh):** = `DC_MWp * effective_yield * (1 - total_loss_pct/100)`. 4. **Year-1 revenue:** = `year1Gen * (price + other)`. Price is ¬£/MWh, other income adds to it. 5. **Degradation:** Assume 0.4% per year (for years 2+). 6. **Cumulative Gen:** Use `calcCumulativeGeneration(dc_mwp, effective_yield, total_loss_pct, degradation_pct, years)` to sum up kWh over 25 or 35 years, as given in code. 7. **Revenue 25yr/35yr:** = `(cumulativeGen * (price + other)) + (BESS25or35)`. BESS returns (if any) are added after solar. 8. **BESS revenue:** If BESS >0, its annual value = `BESS_MWh * cycles * spread * efficiency`. Then `bessAnnual*25` and *35 are added to revenue25 and revenue35. ## CAPEX calculation 1. **Core CAPEX (Wp):** = `EPC_ex + module_cost + other_capex + (floodAdder if applied)`, all in ¬£/Wp. 2. **Base CAPEX (¬£):** = `DC_MWp * 1e6 * core_CAPEX_Wp`. 3. **Contingency:** = `baseCapex * (contingency_pct/100)`. 4. **Fixed CAPEX:** Add fixed CAPEX (site costs, etc.). 5. **BESS CAPEX:** = `BESS_MWh * bess_capex_rate`. 6. **Total CAPEX:** sum of above (base+contingency+fixed+BESS). Exported in ¬£. 7. **CAPEX per Wp:** = `totalCapex / (DC_MWp * 1e6)`. Exported in ¬£/Wp. ## Loss fields (warnings) The fields **DC string loss, LV DC loss, LV AC loss, transformer loss, other loss** are *user-defined*. If any are zero, it might be optimistic. If **all five** are zero, a warning is shown: *‚ÄúSpecialist electrical loss fields are zero. Revenue may be overstated until cable, transformer and electrical design losses are verified.‚Äù* This text is hardcoded in `updateFinanceWarnings()`„Äêcursor‚ÄÝ„Äë. (It matches the developer note exactly.) No source needed, it‚Äôs internal logic. Additionally, warnings triggered by unrealistic values are shown (e.g. negative values, very high loss or bifacial). ## Flood resilience - **Rate vs applied:** The user can check ‚ÄúFlood Resilience‚Äù and enter a rate (¬£/Wp). The model adds this rate to EPC if checked. - In export, we differentiate: - `fin_flood_adder_rate_gbp_wp`: the input rate (available if any). - `fin_flood_adder_applied_gbp_wp`: equals rate if checkbox is ticked, else 0. - `fin_flood_resilience`: boolean of checkbox. - **Rationale:** This makes clear how much was available vs actually used. *Sources:* Solar generation formulas are standard (PVSyst methods, etc.). The specific JavaScript formulas are implemented as shown and can be verified. We cite no external source for arithmetic but note that degeneration and cumulative calc is typical (similar to [Mapbox example code](https://docs.mapbox.com/mapbox-gl-js/api/) style, no direct cite needed). We might cite a solar PV estimation reference, but primary focus is the implementation logic. # Logistics and Packaging Presets The sandbox includes a simple **logistics calculator** to estimate shipping needs from module count. - **Inputs:** The module count (from generation capacity) and chosen packaging preset. - **Presets (generic):** 1. **High-density:** 33 modules per pallet, 594 modules per 40-foot container. 2. **Legacy:** 31 per pallet, 620 per container. 3. **Manual:** User-entered values (not a preset; blank). - **Derived values:** - Modules per pallet and per container are taken directly from presets. - **Total pallets (packing units)** = `ceil(total_modules / modules_per_pallet)`. - **Total containers** = `ceil(total_modules / modules_per_container)`. - **TEU (Twenty-foot equivalent units):** We assume 1 TEU ‚âà 1 pallet capacity (approx 310 modules; provided by container / 2). For export we can either sum up 40-ft containers to TEU by converting (e.g. 594 modules = 1.188 TEU) or better: TEU = `ceil(total_modules / 310)`. The code uses 310 for TEU. - **Ships:** Approx ships = `TEU / 1000` (thousand TEUs per ship), displayed as e.g. `0.05 Ships`. (Exact formula: `ships = (teu/1000).toFixed(2)`). - **Outputs:** These are displayed live in the UI (stat-box) and exported: - `modules_per_pallet_hd`, `modules_per_container_hd`, `total_packing_units_hd`, `total_containers_hd` (for high-density). - Similarly for legacy. - `total_teu`, `dedicated_ships_approx`. We supply the preset values in the user instructions but they are not exported fields per se; only derived totals are. *Sources:* Preset values were given in user instructions.
