Huge File Maintenance and Modularisation Doctrine

GlobalGrid2050 will contain huge files during periods of rapid development.

Huge files are acceptable when they preserve a working product, but they must not become permanent chaos.

The system must support 2 valid paths:

1. Maintain the huge file safely
2. Modularise the huge file safely

Both paths must be handled through Python scripts, YAML manifests and GitHub Actions.

Path 1: Maintain Huge Files Safely

If a large HTML, CSS, JavaScript, Python or GeoJSON file is working, do not break it merely to make it look cleaner.

Large working files may remain intact when:

1. The browser remains stable
2. The file still loads correctly
3. The logic is understandable
4. Feature changes are controlled by manifests
5. GridBot can edit it safely
6. Reports confirm what changed

In this mode, Python performs precise edits inside the large file without rewriting the whole file.

Examples:

Replace heading
Insert new warning block
Add new UI panel
Add export field
Update disclaimer
Inject new script section
Patch calculation logic

This protects the working version while still allowing progress.

Path 2: Modularise Huge Files Safely

When a file becomes too large to maintain directly, it must be modularised by script, not by careless manual cutting.

Python and GitHub Actions must be used to split large files into controlled modules.

Example modularisation targets:

Inline CSS to external CSS file
Inline JavaScript to external JavaScript file
Configuration constants to config file
Financial logic to finance module
Map rendering logic to map module
Drawing logic to topology module
Export logic to export module
Warnings to validation module

The modularisation process must be incremental:

1. Copy working version
2. Extract one section
3. Reconnect it
4. Run checks
5. Commit
6. Test in browser
7. Only then continue

No mass refactor without rollback.

No silent rewrites.

No breaking a working monolith for aesthetic reasons.

Exact Manifest Capabilities

Each manifest.yml may support controlled operations.

Replace operation:

operations:
type: replace
file: path/to/file.html
find: old text
replace: new text
count: 1

Insert after operation:

operations:
type: insert_after
file: path/to/file.html
marker: known marker
insert: new inserted block

Insert before operation:

operations:
type: insert_before
file: path/to/file.html
marker: known marker
insert: new inserted block

Regex replace operation:

operations:
type: regex_replace
file: path/to/file.html
pattern: regex pattern
replace: replacement text
flags: ms

Validation operation:

operations:
type: assert_contains
file: path/to/file.html
text: required validation text

These operations allow very large files to be modified precisely without forcing humans or AI agents to rewrite the full file.
