# GridBot Feature Install Instructions and Progress Log

How to use this document: use this file as the operating instruction for installing controlled features into the GlobalGrid2050 GIS SLD Financial Sandbox V2. To proceed with a feature install, create or update the relevant folder under `feature_requests/`, make sure that folder contains a valid `manifest.yml`, place any new files under the matching `files/` path, commit the feature request to `main`, then go to GitHub Actions, open `GridBot Feature Install`, press `Run workflow`, set `source` to `solar-bess-topology-v2`, set `target` to `solar-bess-topology-v2`, set `feature` to the exact feature folder name such as `002_extract_v2_config`, leave `overwrite` unchecked or false, then run the workflow. Do not leave the feature field blank unless every earlier feature is repeat safe, because a blank feature field tells GridBot to run all feature folders. A specific feature field tells GridBot to run only that one feature. After the workflow completes, check that it is green, open `gridbot_reports/`, confirm that a new install report exists, check the changed files, wait for GitHub Pages deployment to complete, open the live sandbox, confirm that the page renders, the map loads, the substation layer loads, the draw button works, finance values still update and no obvious browser error has appeared. GlobalGrid2050 is being modularised because the GIS SLD Financial Sandbox V2 is already a working browser based infrastructure reasoning system, combining GIS, solar topology, BESS assumptions, grid node selection, substation reference data, cable route awareness, module count logic, CAPEX assumptions, revenue assumptions, electrical loss assumptions, GeoJSON export and early stage project screening. The app began as a large working HTML file at `solar-bess-topology-v2/indexforgis-sld-v2.html`, which was useful for fast development, but it is not the final structure. The goal is not to rewrite the app, and it is not to break a working file into pieces for appearance. The goal is to use GridBot, Python, YAML and GitHub Actions to extract one controlled section at a time while keeping the live app working after every step. A working large file is better than a broken modular system, but a safely modularised system is better than a permanent monolith. The core doctrine is: one feature, one manifest, one controlled change, one GridBot run, one browser test, then continue. Do not manually copy large code blocks into production files when a feature manifest and Python script can perform the change more safely. Do not treat a visual file split as true modularity. True modularity means clear file ownership, clear state ownership, clear load order, clear validation, no hidden dependency breakage, no duplicated logic drifting apart and no silent change to engineering or financial outputs. Feature 003, `003_extract_v2_helpers`, should move only small DOM and formatting helpers such as `$`, `num`, `intVal`, `checked`, `setText`, `setClass`, `money`, `debounce`, `isValidLngLat`, `pickProp` and `setFetchStatus`. It must not move Turf geometry functions, `getRectPolygon` or `getBlockAspect`, because those belong later in topology. Feature 004, `004_create_test_fixtures`, should create known good GeoJSON output fixtures before changing finance, topology or export logic. Fixture cases should include an empty or zero case, a default string case, a default central case, a BESS active case and a minimum edge case. This matters because manual smoke testing is not enough for an app that produces technical and financial outputs. Before deeper refactoring, the current outputs should be captured so future changes can be checked against them. Feature 004a, `004a_fixture_harness`, should create a simple runner to compare future outputs against committed fixtures, because fixtures alone are not enough. The harness must provide a green or red signal after future GridBot installs and becomes the safety net before finance, topology and export are changed. Feature 005, `005_decouple_finance_inputs`, should separate finance input reading from finance calculation. The current issue is that `computeFinance(prefix, stats)` reads many DOM inputs directly, meaning finance is not yet pure calculation logic. The correct approach is to create `readFinanceInputs(prefix)`, which returns a plain input object, then update `computeFinance(inputs, stats)` so it accepts plain data instead of reading DOM directly. BESS efficiency units should also be cleaned here so the input object uses one consistent representation. Feature 006, `006_extract_v2_finance`, should move finance calculation into `gis-sld-v2-finance.js`, but only after Feature 005 proves finance input decoupling works. Finance calculation should not directly depend on document elements. Feature 007, `007_extract_v2_state`, should create clear state ownership. The current app has a central state object, and before map, topology and export are split, the project must decide whether state remains a shared object or moves into a namespace such as `window.GISSLD`. This must happen before map and topology extraction. Feature 008, `008_extract_v2_map`, should move MapLibre related logic into `gis-sld-v2-map.js`, including map setup, layer setup, popup logic, substation loading, basemap toggles and map click handlers.
