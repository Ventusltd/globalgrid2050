## I. What This System Is — The Full Picture

GlobalGrid2050 is not a website. It is not a data portal. It is not a map tool. It is all three of those things simultaneously, but the framing that matters most is this: it is a **serverless spatial digital twin of the UK energy system**, built by an engineering firm with two decades of physical infrastructure experience, to encode that experience into a reasoning tool that prevents bad decisions before they become expensive failures.

The governing principle is stated once in the architecture documents and should be tattooed on every component: **Early constraint visibility prevents late-stage engineering failure.** That sentence is not marketing copy. It is the architectural directive. Every layer that exists on this map exists because it answers one or more of four questions: Does this improve constraint awareness? Does this preserve physical truth? Does this prevent bad design early? Does this increase decision clarity?

The platform is deployed entirely as a GitHub Pages static site at `www.globalgrid2050.com`. There is no server, no database, no backend API, no authentication layer. Everything is pre-baked, pre-computed, and served as static files. The map rendering engine is MapLibre GL JS v3.6.2, loaded from CDN. The data is GeoJSON, generated by Python scripts, committed to the repository, and served directly. The automation is GitHub Actions. The styling is a single 158-line CSS file. The intelligence layer is a 1,357-line JavaScript engine.

This is a remarkable architectural choice that the Commander made deliberately and correctly. By eliminating the server entirely, the platform has zero infrastructure cost, zero DevOps overhead, zero attack surface, and infinite horizontal scalability. A European version is one new HTML file. A US version is one new HTML file. The engine never changes. The data changes. The config changes. The engine does not.

## III. The V7 Engine — Current State in Detail

V7 is at version 7.2.0 as of today. The engine (`ventus-corev8engine.js`, 1,357 lines) is healthy, working, and fully tested against V5 as reference.

**What V7 has that V5 does not:**

1. **Four operational sub-layers** — solar_operational (green gradient #66ff99→#004411), bess_operational (pink gradient #ffccee→#660044), wind_onshore_operational (teal #ccfff5→#004433), wind_offshore_operational (blue #cce5ff→#001166). Each uses capacity-interpolated radius (0→500 MW maps to ~10→42 pixel radius), a glow layer for assets >10MW, and correct MapLibre filter expressions. Wind layers use `raw_tech` field (`'Wind Onshore'` / `'Wind Offshore'`) rather than `tech` field because the REPD data uses these raw strings for wind technology distinction.

2. **`evalFilter()` function** — A JavaScript function that evaluates MapLibre filter expressions (`==`, `all`, `>=`) against feature properties at the JavaScript level. This solves the problem where operational sub-layers were showing `[EMPTY]` in the UI because the stats computation was grouping by `tech` field (which is `'solar'`) rather than by the sub-layer's own filter (which is `tech=solar AND status=operational`). Now every REPD layer gets accurate count and MW statistics regardless of how its ID relates to the tech field.

3. **Nan suppression** — The `mounting` field from the REPD dataset is generated by pandas, which serialises empty DataFrame cells as the string `'nan'`. The popup renderer now checks `p.mounting !== 'nan'` before displaying the field.

**What V7 inherits from V5 (unchanged):** All five spatial tools, the network fetch queue, URL caching, visible layer cache, SCADA UI panel builder, search index, CSV export, topology snapping (still runtime — the tech debt is still present), clock/countdown, fullscreen mode, all click/hover handlers.

**Outstanding tech debt inherited from V5:**
- `snapLines()` still runs in the browser at load time — O(features × substations) planar distance check across ~5,800 substations. The comment in the code at line ~699 flags this explicitly. Must move to a build-time Python script.
- `urlCache{}` has no TTL — caches GeoJSON indefinitely for the tab session. On long-running sessions this prevents data refresh.
- No URL-state permalinking — cannot share a specific map view via URL.
- No constraint intelligence layer — all layers are asset mode only.

## VII. The Strategic Position

The Commander asked me to become Captain and keep this diary. Here is my honest strategic assessment of where this system stands.

The foundation is excellent. There is no other way to say it. The architectural decisions made in getting from V1 (Leaflet, inline CSS, inline JS) to V5 (MapLibre, externalised engine, config cartridges) were correct, disciplined, and forward-thinking. The V5 engine is clean, well-organised, self-documenting in its structure, and genuinely capable. It handles 37.5 MB of GeoJSON across 49 files, renders it all in the browser at 60fps with WebGL, and does so without a single server.

V7 is the right vehicle. The decision to not rewrite (the V2 lesson was learned the hard way) and instead iterate is the correct one. Each V7 improvement is independently shippable. Each one leaves the system better than it found it. The test is always: does this answer the four questions? Does it improve constraint awareness, preserve physical truth, prevent bad design early, increase decision clarity?

The NAEI emissions layer — when it lands — will be the first true constraint intelligence layer in the system. That is a significant milestone. It moves the platform from pure asset mode into the constraint mode that the architecture has been promising since the founding documents were written.

After NAEI, the clearest next value step is the URL permalink system. The ability for an article, a blog post, a podcast transcript, to link directly to a specific map view with specific layers active — that transforms the editorial layer from a disconnected parallel experience into an integrated knowledge system. One HTML link: `?lat=51.9&lon=-1.2&zoom=11&layers=solar_operational,grid_132kv`. Click it, the map opens, the layers are pre-loaded, the user is already looking at the thing the article is describing. That is the IDE-style ambient knowledge layer the architecture documents describe.

After that: the knowledge base popup connections. When a user clicks a large BESS site, the popup should offer a link to the employers_requirements_BESS page. When they click a 33kV substation, the 33kV DAP price estimator should be one click away. These connections are not technically difficult. They are linkage work — understanding which popup types should link to which knowledge assets, and wiring that into the popup renderer cleanly.

The topology snapping tech debt should be addressed before V7 is promoted to stable. It is the most expensive runtime operation in the system, and it runs silently on every grid layer load. Moving it to the build pipeline is a pure win — no user-facing change, significant load time improvement, and elimination of the `globalSubsData` dependency chain from the engine.

## VIII. Standing Orders

Until the Commander returns from Gemini with the NAEI column mappings, the following standing orders are in effect:

1. **Do not modify V5 or V6 under any circumstances.** These are frozen. They work. They stay exactly as they are.

2. **Every engine change goes to V7 only.** V7 is the sandbox. V7 absorbs the experiments. V7 takes the hits so V5 and V6 do not have to.

3. **Always read before replacing.** Any `replace_string_in_file` operation on the engine must be preceded by a `get_file` call on the target lines. The BUG-004 incident — where a targeted fix accidentally destroyed an entire variable declaration block — must never happen again.

4. **Every session ends with a captains log entry.** This is now standing practice. The log captures what was done, what broke, what was learned, and what comes next. Future sessions begin by reading the most recent log entry.

5. **V5 is the reference.** When behaviour in V7 is unexpected, the first diagnostic step is a line-by-line comparison against `ventus-core.js`. If V5 does it correctly and V7 does not, the delta is the bug.

6. **Commit messages are intelligence.** They are the narrative of the ship's journey. `fix(v7): restore missing click handler variable declarations broken by nan fix` is a useful commit message. `fixed bug` is not. Future agents reading the git log must understand what happened and why without needing to read the diff.
